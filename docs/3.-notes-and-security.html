3. Notes And Security

This page exists because “secure” is a loaded word. Here’s the unfiltered truth about what Secure Store actually protects, and what it deliberately does not try to do.

Threat Model Protected? How / Why
Other non-admin local users on the same machine Yes Windows: explicit ACL grants only current user SID Linux/macOS: chmod 600 (owner-only)
Malware running as a different user account Yes Same OS-enforced boundary as above
Casual snooping in /tmp or %LocalAppData% Yes Uses per-session $XDG_RUNTIME_DIR when available; otherwise still forces 600 permissions
Malware or another app running as the same user No Impossible to stop with just filesystem permissions. Same-user processes have identical rights.
Process injection, .NET hooking, memory scraping No Secure Store is just JSON files. Once your process is compromised, the data is already in memory.
Administrator / root No Admin can read everything anyway (by design of the OS).
Physical access or cold-boot attacks No No encryption at rest yet. Relies purely on filesystem isolation.

Real-World Security Role in XRUIOS / Plagues Protocol

Secure Store is not the final security boundary. It’s the hand-off layer.

Secure Store Was Originally Made For Plague Worker

Here’s how it actually gets used in production:

  1. Untrusted UI process (your app, plugins, shell) launches.
  2. It needs to talk to a privileged Plagues Worker running as SYSTEM/root.
  3. The worker starts first (at login), generates a random local gRPC endpoint (e.g. http://127.0.0.1:5057), and writes it to Secure Store with SecureStore.Set("worker_addr", endpoint).
  4. The untrusted UI later does SecureStore.Get("worker_addr") → connects → now all real secrets live only inside the signed, manifest-verified, privileged worker.
  5. Secure Store never holds anything truly sensitive longer than a few seconds.

Result: Even if the entire userland is compromised, the attacker only ever learns a temporary local port number — which is useless without also compromising the Plagues Worker (which self-verifies its own executable + all DLLs with Kyber-signed Blake3 manifests before listening).

Bottom Line

Use Case Safe with Secure Store alone? Need full Plagues Worker?
Storing local gRPC/named-pipe endpoints Yes Recommended
Temporary session tokens, window positions, themes Yes No
Refresh tokens, API keys, passwords No Yes (must go in worker)
Anything you would cry about if leaked No Yes

Secure Store = “secure enough to survive until the real fortress (Plagues Worker) wakes up”.

That’s by design. Small, auditable, zero dependencies, works everywhere. Anything more belongs behind the cryptographically-sealed boundary of a Plague Worker.
